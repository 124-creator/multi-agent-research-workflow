V4 流程图

以最新工作流图片为准：轨道选择器 → Loop 1 方案层 → Loop 2 协同交付层 → 状态机与全局不变量。

轨道选择器

第一步不是开干，而是判断复杂度。小任务允许塌缩；不确定性越高，门禁越完整。

轻型

一句话可描述 diff

局部修改、路线明确、返工代价低。使用轻量任务卡 → 修改 → 最小测试 → 执行回执。

标准

多文件功能 / 需比较路线

存在 2–3 种合理做法，需要完整 Loop 1 + Loop 2，012 可选，Spike C 建议做。

重型

高不确定 / 高风险

科研、Agent、外部系统集成、安全隐私合规、重大性能风险。012 与 Spike A/B/C 全做。

Loop 1 · 方案层：先做对的事

目标是冻结问题、建立证据、比较路线、对抗评审、裁决路线并用 Spike 验证最危险假设。

010

问题定义与需求确认

冻结一句话目标、成功标准、硬约束、明确不做和验收口径；FROZEN 前必须人类逐条确认。

G1 第一道：方向是否值得做
012

难题风险地图

先写清楚难在哪、关键未知项、风险类型和升级条件；重型必做，标准可选，轻型跳过。

防止过早收敛
015

调研基线与证据清单

GPT-Pro 建立外部调研基线；每条关键证据标 A/B/C/D，区分 FACT / INFERENCE / UNKNOWN。

证据 ID：L1-E###
020

中性候选路线

生成 2–3 条候选路线，列出适用条件、失败模式、证据映射和最小验证方式。

不先选路线再找证据
030

独立对抗评审

Claude 网页端独立联网证伪，输出 BLOCKER / RISK / 最小缓解方案；BLOCKER 必须通过校准表。

反方不是顺从式润色
035

路线裁决

新鲜上下文处置所有 BLOCKER，明确 SELECTED 路线、未选原因、保留风险和 Spike 设计。

G1 第二道：路线确认
040

三类验证桩

Spike A 可行性、Spike B 反证、Spike C 集成。必做段 FAIL 或无证据则返回路线阶段。

进入 Loop 2 的唯一入口

Loop 2 · 协同交付层：再把事做对

Claude Code 负责规划、派工和终检；Codex 只执行 approved plan，并用证据、测试和复盘交付 implemented。

08

Claude Code 仓库探索 + 联网实现调研

从 040 接手，按回链读取 010/012/015/020/030/035，建立同号调研台账。

证据 ID：CC-R###
09

INSTRUCTION LOAD CHECK

验证关键指令真的被加载，检查 override、截断、hook 状态和上下文遗漏。

未通过不得规划/执行
10

draft plan + Test Oracle + PLAN AUDIT

写入 docs/dev/plans/NNN-模块名-计划.md；产物清单、验收锚点、测试路径、回滚边界必须齐备。

没有 Test Oracle 不得 approved
11

G2 执行门

人类批准 plan 从 draft 变为 approved；外部写操作是 G2 内的子决策，必须逐项批准。

批准“怎么做”
12

WORKTREE CHECK

标准/重型硬门禁：检查文件隔离，也检查 DB、环境变量、端口、服务和资源隔离。

防止污染运行项目
13

Codex 独立联网复核 + 实现 + 测试

Codex 不改计划、不换路线；若证据冲突则 RESEARCH CONFLICT 回传，否则按清单落盘并测试。

状态推进到 implemented
14

Claude Code 终检

新鲜上下文 + 联网反证 + CC-V 证据；核查产物命名、规格符合性、Test Oracle 和安全边界。

INITIAL REVIEW: PASS/REWORK/BLOCKED
15

G3 验收门

人类用新鲜证据终验，确认 verified。verified 只能由人类终验产生，不能由执行自述产生。

批准“是否真的完成”

状态机

停机不能伪装成完成；blocked 就是 blocked。

draft→approved→implemented→verified

execution: not_started → researching → running → blocked | completed
review: pending → rework | blocked | pass

三道人类门

G1 方向门：010 FROZEN + 035 SELECTED，确认“去哪”。
G2 执行门：plan approved，确认“怎么做”。
G3 验收门：verified，确认“是否真的完成”。

文件与证据总线

角色之间不搬聊天记录，而是通过同号文件、证据 ID 和回链传递上下文。

Loop 1 Spec

docs/dev/spec/010、012、015、020、030、035、040。040 是 Loop 2 唯一入口。

同号交付链

research/NNN-调研台账 → plans/NNN-计划 → retrospectives/NNN-复盘 → reviews/NNN-核查。

证据 ID

L1-E###、CC-R###、CX-R###、CC-V###；关键结论必须引用证据 ID。

全局红线

  • 禁止 标准/重型任务未通过 010/015/035/040 就进入 Loop 2。
  • 禁止 执行未 approved 的 plan；没有 PLAN AUDIT: PASS 不交接。
  • 禁止 没有 Test Oracle / 验收锚点的 plan 被 approved。
  • 禁止 Codex 自行换路线、改范围、改验收或降低测试来宣称通过。
  • 禁止 上传私有代码/数据、硬编码密钥、执行未知脚本。
  • 要求 标准/重型通过 WORKTREE CHECK，执行前通过 INSTRUCTION LOAD CHECK。

复制提示词入口

这不是完整指令文件，只是启动入口。正式执行仍以仓库 docs/v4 下的三份 V4 指令为准。

请按 ResearchLoop V4 启动本任务。
第一步只做轨道选择与 010 问题定义：
1. 判断任务是轻型 / 标准 / 重型，并说明理由；
2. 写出一句话目标、成功标准、硬约束、明确不做、已知事实、未知项；
3. 对成功标准和验收口径逐条向人类确认；
4. 未 FROZEN 前不得进入路线选择或实现。
你是 ResearchLoop V4 的 Claude Code 审查官。
请先读取 docs/v4/000-dual-loop-controller-v4.md 与 010-claude-code-reviewer-v4.md。
从 docs/dev/spec/040-验证桩结论.md 接手，按回链读取 010/012/015/020/030/035。
完成 INSTRUCTION LOAD CHECK、仓库探索、联网实现调研、同号调研台账、draft plan、Test Oracle 与 PLAN AUDIT。
未通过门禁不得写业务实现；plan 未 approved 不得派工给 Codex。
你是 ResearchLoop V4 的 Codex 执行员。
请先读取 docs/v4/000-dual-loop-controller-v4.md 与 020-codex-executor-v4.md。
只执行用户指定的 approved plan；开始前完成 INSTRUCTION LOAD CHECK、执行前联网复核和 WORKTREE CHECK。
不得改计划、换路线、扩大范围或降低验收。
如发现证据冲突，输出 RESEARCH CONFLICT 并停机回传；如通过，则按产物清单逐字落盘、测试、追加 CX-R 证据和过程复盘，最后输出 EXECUTION DONE。
